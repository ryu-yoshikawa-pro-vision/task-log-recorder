# 設定画面・重複判定・履歴管理の改善計画

## Summary
- ショートカット実行時の通知は、通常の Web ページ上で確実にスナックバー表示される経路に整理し、失敗時も保存結果だけは維持します。
- 設定画面は 2 カラム前提の崩れを解消し、ショートカットカードを全幅化、使い方は縦積みレイアウトへ変更します。
- 重複除外秒数は「同一 `page_url` なら重複」とする仕様に合わせて、ポップアップ記録とショートカット記録の両方で同じ判定に統一します。
- 履歴タブに「すべて削除」を追加し、確認ダイアログ後に `logs` のみ削除します。`lastFingerprint`、`lastTaskName`、`shortcutStatus` は保持します。
- Playwright CLI は実画面確認に使います。まず設定画面と履歴 UI を自動確認し、ショートカット通知は拡張機能付き起動で再現できる範囲まで検証します。

## Implementation Changes
- 設定画面
  - `settings-grid` を「基本 / コピー」2 カラム + 「ショートカット」全幅 + 「使い方」全幅の構成に整理する。
  - CSS でショートカットカードの全幅指定、右側空白が出ないグリッド調整、`help-grid` の 1 カラム化、モバイル時の縦積みを明示する。
  - 「基本の流れ」「補足」は常に上下配置にし、読み順が左上から自然になるようにする。
- ショートカット通知
  - 通知処理を、アクティブタブ解決とページ通知失敗時の扱いが追いやすい形に整理する。
  - 保存成功時と重複警告時の通知メッセージ生成を維持しつつ、ページ通知が失敗しても記録処理自体は継続する。
  - 通知対象は通常の Web ページを前提にし、`chrome://` など注入不可ページでは従来どおり保存とステータス更新のみ行う。
- 重複判定
  - 重複キー生成を「同一 `page_url`」基準に変更し、`event_type`・`task_name`・`memo` 差分は無視する。
  - popup と shortcut の両方で同じ重複キーを使う。
  - URL 比較は保存されている `page_url` の文字列一致をデフォルトとする。
- 履歴全削除
  - 履歴ヘッダーに「履歴をすべて削除」ボタンを追加する。
  - confirm 付きの全削除処理を追加し、削除後はページネーション、メトリクス、TSV プレビューを再描画する。
  - `logs` 全削除用ヘルパーを追加する。削除対象は `logs` のみ。

## Interface / State Changes
- `logs` 全削除用の storage API を追加する。
- 重複判定用キーの生成ロジックを変更する。
- UI 追加要素:
  - 設定画面の全幅ショートカットセクション
  - 履歴タブの「履歴をすべて削除」ボタン
  - 削除確認ダイアログはブラウザ標準 confirm を使う

## Test Plan
- 単体テスト
  - `tests/utils.test.js` に「同一 URL ならイベント種別が違っても重複になる」ケースを追加する。
  - `tests/shortcut.test.js` に URL-only 重複判定と通知継続の期待値を追加する。
  - `tests/storage.test.js` に `logs` 全削除ヘルパーの追加テストを入れる。
- 静的検証
  - `npm test`
  - `npm run lint`
- 実画面確認
  - Playwright CLI の headed 実行で設定画面を開き、ショートカットカード全幅化と「使い方」の縦並びを確認する。
  - ポップアップで履歴複数件投入後、「すべて削除」確認ダイアログ、削除後の空状態、TSV プレビュー初期化を確認する。
  - 拡張機能を載せた Chromium セッションでショートカット通知を再現できる場合は実施し、難しい場合はここだけ手動 Chrome 確認に切り替える。

## Assumptions
- 重複除外秒数は、指定秒数内の「同一 `page_url`」だけを重複とみなす。
- 履歴全削除は `logs` のみ対象で、`lastFingerprint`・`lastTaskName`・`shortcutStatus` は残す。
- ショートカットのスナックバーは注入可能な通常 Web ページでの表示を対象とし、Chrome 内部ページは対象外とする。
